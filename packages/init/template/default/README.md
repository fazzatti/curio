## Admin

The app mounts the server-rendered Curio admin at `/admin`.

Out of the box:

- `admin:create` bootstraps the first `superadmin`
- `/admin/login` signs into the admin with email and password
- `/admin` shows a widget-driven dashboard
- `/admin/resources/users` supports list, detail, create, edit, reset password,
  and guarded delete
- `/admin/resources/roles` and `/admin/resources/permissions` are editable
  built-in resources for `superadmin`
- `/admin/resources/sessions` and `/admin/resources/audit` remain read-only
  built-in resources
- custom admin pages can live under `/admin/views/...`
- custom admin workflows can live under `/admin/flows/...`

Important admin design rules:

- the admin is mounted directly in the backend, not as a separate frontend app
- the admin talks to Curio DB repositories directly, not through the public API
- admin auth uses DB-backed cookie sessions
- RBAC is explicit and normalized
- `superadmin` is a bypass role
- the default admin preset is mounted with `presets: ["default"]`
- password resets revoke the target user's active admin sessions

## Superadmin Bootstrap

Run the interactive command:

```sh
deno task admin:create
```

The command prepares the schema if needed, seeds built-in admin data, prompts
for credentials, creates the first `superadmin`, and refuses to create a second
one.
